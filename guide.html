What You Will Build

Fully working observability stack running on two Linux servers, all deployed automatically with a single Ansible command.

Goal
Run ansible-playbook playbook.yml and get a live Grafana dashboard showing real CPU, memory, disk, and network metrics from two servers.
Deployment Defaults
The provisioning script in monitoring/scripts/provision_ec2.sh defaults to AWS region us-east-2. You can change the region by setting the environment variable AWS_DEFAULT_REGION or when prompted by the script.

System Architecture
Prerequisites
A
Two EC2 instances (Ubuntu 22.04)

Launch two Ubuntu 22.04 EC2 instances (a monitoring server and a target server). They should be reachable from your workstation via their public IPs or be in the same VPC if using private addresses.

B
Ansible Control Node

This is where you install Ansible and run playbooks. It only needs SSH access to the two servers.

C
EC2 Key Pair / SSH Access

When launching EC2 instances, create or select an EC2 Key Pair (a .pem file). Use that private key to SSH: ssh -i ~/keys/my-ec2-key.pem ubuntu@<public-ip>. Ensure the key file has strict permissions: chmod 600.

Install Ansible on Your Control Node

Run these commands on your local machine

bash — control node 
$ # Ubuntu / Debian
sudo apt update
sudo apt install -y software-properties-common
sudo add-apt-repository --yes --update ppa:ansible/ansible
sudo apt install -y ansible

# Verify installation
ansible --version
3 Set Up SSH Key Authentication

Ansible communicates with servers over SSH. We need passwordless login.

bash — control node 
# If you launched EC2 instances with a key pair (.pem)
  chmod 600 ~/keys/my-ec2-key.pem
  ssh -i ~/keys/my-ec2-key.pem ubuntu@<monitoring-public-ip>
  ssh -i ~/keys/my-ec2-key.pem ubuntu@<target-public-ip>

  # Alternative: If you need to add your own public key to the instance (for a different key)
  ssh -i ~/keys/my-ec2-key.pem ubuntu@<public-ip> 'mkdir -p ~/.ssh && echo "$(cat ~/.ssh/id_rsa.pub)" >> ~/.ssh/authorized_keys'

  # Test SSH connections (should NOT ask for password when using the PEM key)
  ssh -i ~/keys/my-ec2-key.pem ubuntu@<monitoring-public-ip> "echo 'monitoring OK'"
  ssh -i ~/keys/my-ec2-key.pem ubuntu@<target-public-ip> "echo 'target OK'"
⚠️ Common Issue
If ssh-copy-id fails, check that the ubuntu user exists on the server and password authentication is temporarily enabled in /etc/ssh/sshd_config (PasswordAuthentication yes).
4 Set Up the Project Files

Download or unzip the project folder, then update it with your server IPs.

bash — control node 
# Unzip the project
unzip monitoring-ansible.zip
cd monitoring

# Verify the folder structure
tree .

Edit inventory.ini

Open inventory.ini and replace the IP addresses with your actual server IPs:

inventory.ini 
[monitoring_server]
  monitoring ansible_host=<MONITORING_PUBLIC_IP> ansible_user=ubuntu ansible_ssh_private_key_file=~/keys/my-ec2-key.pem

  [target_servers]
  target ansible_host=<TARGET_PUBLIC_IP> ansible_user=ubuntu ansible_ssh_private_key_file=~/keys/my-ec2-key.pem

  [all:vars]
  ansible_python_interpreter=/usr/bin/python3

  # Tip: replace <MONITORING_PUBLIC_IP> and <TARGET_PUBLIC_IP> with the public IPs from the EC2 console.

Edit group_vars/all.yml

Update the target_server_ip variable to your target server's IP. Prometheus needs this to know where to scrape metrics from.

group_vars/all.yml — key lines 
# ↓ Change this to your actual target server IP (use the EC2 private IP if Prometheus scrapes over VPC)
  target_server_ip: "<TARGET_PRIVATE_IP or TARGET_PUBLIC_IP>"

  # Optional: change Grafana password
  grafana_admin_password: "admin123"

  # If monitoring uses private VPC addresses, use the target's private IP here and ensure security group allows traffic between the monitoring and target instances.
5 Understanding the Project Structure
🧠 Ansible Concept: Roles
A role is a reusable, self-contained unit of automation. Instead of writing one huge playbook, we split work into roles: node_exporter, prometheus, and grafana. Each role has its own tasks, templates, and default variables. This is how professional Ansible code is organized.
Project tree 
monitoring/
├── inventory.ini          ← Defines which servers to manage
├── ansible.cfg            ← Ansible settings (user, SSH, etc.)
├── playbook.yml           ← Master playbook (entry point)
├── group_vars/
│   └── all.yml            ← Variables shared across all hosts
└── roles/
    ├── node_exporter/
    │   ├── tasks/main.yml   ← Install Node Exporter steps
    │   └── defaults/main.yml ← Default variable values
    ├── prometheus/
    │   ├── tasks/main.yml   ← Install Prometheus steps
    │   └── templates/
    │       └── prometheus.yml.j2  ← Config template with variables
    └── grafana/
        ├── tasks/main.yml   ← Install Grafana steps
        └── templates/
            └── grafana.ini.j2  ← Grafana config template
🧠 Ansible Concept: Jinja2 Templates (.j2)
Template files use {{ variable_name }} syntax. When Ansible runs, it replaces these placeholders with actual values from group_vars/all.yml. This means one template works for any environment — just change the variables!

The Prometheus Config Template (rendered output)

prometheus.yml — what gets deployed to server 
global:
  scrape_interval:     15s
  evaluation_interval: 15s

scrape_configs:

  # Prometheus monitors itself
  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']

  # Scrape local Node Exporter (monitoring server)
  - job_name: 'node_exporter_monitoring'
    static_configs:
      - targets: ['localhost:9100']
        labels:
          instance: 'monitoring-server'

  # Scrape remote Node Exporter (target server)
  - job_name: 'node_exporter_target'
    static_configs:
      - targets: ['192.168.1.20:9100']
        labels:
          instance: 'target-server'
6 Run the Playbook

Now the exciting part. One command deploys everything:

bash — control node (from inside the monitoring/ folder) 
# First: do a dry run (no changes, just check)
ansible-playbook playbook.yml --check

# Run for real
ansible-playbook playbook.yml

# Run with verbose output (great for learning/debugging)
ansible-playbook playbook.yml -v

# Run only a specific role using tags
ansible-playbook playbook.yml --tags node_exporter

What You'll See (expected output)

playbook output 
PLAY [📦 Install Node Exporter on all hosts] *****

TASK [node_exporter : Create node_exporter system user]
ok: [monitoring]
ok: [target]

TASK [node_exporter : Download Node Exporter binary]
changed: [monitoring]
changed: [target]

...

TASK [node_exporter : Print Node Exporter status]
ok: [monitoring] => {
    "msg": "✅ Node Exporter is UP on port 9100"
}

PLAY RECAP *****
monitoring  : ok=12  changed=8   unreachable=0  failed=0
target      : ok=7   changed=5   unreachable=0  failed=0
🧠 Ansible Concept: Idempotency
Notice ok vs changed in the output. If you run the playbook again, tasks that are already in the desired state show ok (no changes made). This is idempotency — you can safely run the playbook 100 times and it will only change what needs changing. This makes automation safe and predictable.

# Check Node Exporter status on all hosts
ansible all -m command -a "systemctl is-active node_exporter"

# Check Prometheus on monitoring server
ansible monitoring_server -m command -a "systemctl is-active prometheus"

# View Prometheus config that was deployed
ansible monitoring_server -m command -a "cat /etc/prometheus/prometheus.yml"
8 Explore the Prometheus UI

Open your browser and go to: http://<monitoring_server_ip>:9090

1
Check Targets

Click Status → Targets. You should see 3 targets all showing State: UP — prometheus, node_exporter_monitoring, and node_exporter_target.

2
Run a Query

In the search box, type node_cpu_seconds_total and click Execute. You'll see raw CPU metrics from both servers.

3
Try a PromQL expression

Type 100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) — this gives you live CPU usage % per server.

9 Set Up Grafana Dashboard

Open your browser and go to: http://<monitoring_server_ip>:3000

🔑 Default Login
Username: admin    Password: admin123 (set in group_vars/all.yml)
Grafana will prompt you to change the password on first login — you can skip this for the lab.
1
Verify the Prometheus data source

Go to Connections → Data Sources. You should already see Prometheus listed (provisioned by Ansible). Click it and hit Save & Test to confirm it connects.

2
Import the Node Exporter Full dashboard

Click the + icon in the sidebar → Import dashboard. In the "Import via grafana.com" field, enter: 1860. Click Load.

3
Select your data source

On the next screen, under the Prometheus dropdown, select Prometheus. Click Import.

4
Explore your live dashboard!

You should now see a rich dashboard with CPU usage, memory, disk I/O, network traffic, load average, and more — live from both servers. Use the instance dropdown at the top to switch between servers.

✅ Pro Tip — Dashboard ID 1860
The "Node Exporter Full" dashboard (ID 1860) is one of the most popular community dashboards on grafana.com. It was designed specifically for Node Exporter and works perfectly with our Prometheus setup.

"Today we're going to deploy an entire monitoring stack — three production-grade tools — across two servers. We'll do it with a single command, and in about 5 minutes. This is the power of Infrastructure as Code."

Closing Key Points

★
Idempotency

Safe to run multiple times. Ansible only changes what needs to change. No accidental restarts.

★
Declarative Infrastructure

You describe the desired state ("Grafana should be running") not the steps to get there. Ansible figures out the steps.

★
Version Control

All these files live in Git. Your entire infrastructure is documented and reproducible. "If it's not in Git, it doesn't exist."

★
Scaling

Need to monitor 100 servers? Add them to inventory, update prometheus config, run one command. Done.

11 Troubleshooting Guide
ProblemLikely CauseFix
UNREACHABLE! in playbook SSH not configured Check ssh-copy-id and test manual SSH
Target shows DOWN in Prometheus Firewall blocking port 9100 Run ufw allow 9100 on target server
Grafana shows "Data source error" Prometheus not running systemctl restart prometheus
Port 3000 not accessible from browser Firewall on monitoring server ufw allow 3000 and ufw allow 9090
Failed to connect to the host Wrong IP in inventory Double check inventory.ini IPs
Download tasks timeout Server has no internet Check curl https://github.com from server
bash — useful debug commands 
# Test Ansible can reach all hosts
ansible all -m ping

# Check Node Exporter logs
journalctl -u node_exporter -n 50 --no-pager

# Check Prometheus logs
journalctl -u prometheus -n 50 --no-pager

journalctl -u grafana-server -n 50 --no-pager

# Validate Prometheus config
promtool check config /etc/prometheus/prometheus.yml

# Reload Prometheus config without restart
curl -X POST http://localhost:9090/-/reload
